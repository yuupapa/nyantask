## 2. 解決する課題

### コンサル生が継続できない本質的な理由
- YouTubeの成果は遅効性（投稿→結果に数週間〜数ヶ月）
- 日々のタスクと成果が体感で繋がらない
- サボっても今日は何も困らない（即時のペナルティがない）
- やっても今日は何も変わらない（即時のリワードがない）

### 仮説
「即時の感情報酬＋即時の損失回避」を猫の育成で作れば、成果が出るまでの"空白期間"を埋められる。

## 3. ゲーム設計（3層ループ）

| ループ | サイクル | 行動 | 報酬 |
|--------|----------|------|------|
| 日次 | 毎日 | タスク完了→おさかな獲得→餌・遊び | 猫が喜ぶ、満腹ゲージ回復、なかよしLv UP |
| 週次 | 7日単位 | 連続達成ストリーク維持 | 成長段階が進む、にくきゅうボーナス |
| 月次 | 30日単位 | 累積達成でマイルストーン | 新しい猫の解放、レア猫の出現 |

### 猫の状態遷移（5段階）

```
ごきげん 😸 → ふつう 🐱 → しょんぼり 😿 → 体調不良 🤒 → 家出 🚪
  ↑ タスク完了で回復                    放置で悪化 ↓
```

「死亡」ではなく「家出」を採用（罪悪感での離脱防止 + 復帰導線確保）。

## 4. 通貨システム（v5新規）

2種類の通貨で「日常的な達成感」と「特別な達成感」を分離します。

### 🪙 コイン（日常通貨）

| 表示 | ホーム画面右上に常時表示 |
| 経済バランス | インフレしてもいい消費型通貨 |

### 🛒 ショップ（Phase 2 実装予定）

コインを消費してアイテムを購入する機能。

#### 餌（満腹度回復）
| グレード | 名前 | 価格 | 満腹度 |
|---------|------|------|--------|
| 梅 | ふつうのエサ | 3コイン | +20 |

#### インテリア（見た目のみ、効果なし）
- 猫のいる部屋・背景の装飾アイテム
- 例：クッション、キャットタワー、窓枠など

#### おもちゃ（効果あり）
- 猫に使うと機嫌UPなどの効果

## 5. なかよしLv.（v5新規）

成長ステージとは**別軸**の親密度メトリクス。長期エンゲージメントの中核。

### 二層構造

| 軸 | 用途 | 進行ロジック |
|---|------|------------|
| **成長ステージ**（5段階） | 見た目の変化、30日でレジェンド | 日数ベース（達成率で停滞/退行あり） |
| **なかよしLv.**（無限） | 親密度、特別アクション解放 | タスク完了の累積経験値ベース |

### Lv UP 報酬
- にくきゅう +3
- 5レベルごとに新しい「なでなでアニメーション」解放
- 10レベルごとに「特別な会話」解放（Gemini API活用）
- 25レベルでその猫専用の称号付与

### なぜ二層にするのか
- 成長ステージは30日でレジェンド到達 → 短期目標
- なかよしLv.は無限 → レジェンド後も継続する動機
- 「成長は止まったけど絆は深まる」体験

## 6. 猫キャラクター設計

### 3要素 × レアリティ階層

```
猫 = 見た目(柄) × 顔(表情パターン) × 性格
```

| 要素 | パターン数 | 例 |
|------|----------|-----|
| 見た目（柄） | 12種 | 三毛・黒・白・キジトラ・サバトラ・茶トラ・ハチワレ・ぶち・シャム・ロシアンブルー・スコティッシュ・マンチカン |
| 顔（表情） | 8種 | まんまる・つり目・たれ目・しょぼ目・笑い目・キリッ目・ジト目・キラキラ目 |
| 性格 | 10種 | おっとり・ツンデレ・甘えん坊・やんちゃ・マイペース・ビビり・寝坊助・くいしんぼ・優等生・天才肌 |

**12 × 8 × 10 = 960通り** + シークレット猫

### レアリティ

| レアリティ | 出現率 | 特徴 |
|-----------|--------|------|
| ★1 ノーマル | 60% | 基本の組み合わせ |
| ★2 アンコモン | 25% | 珍しい柄or性格 |
| ★3 レア | 10% | 特殊な組み合わせ |
| ★4 エピック | 4% | 高難度達成で解放 |
| ★5 シークレット | 1% | 隠し条件のみ |

### 性格の影響
- 成長スピード（おっとりは遅め、やんちゃは早め）
- 家出耐性（ビビりは家出しやすい、優等生は粘り強い）
- なかよしLv UP速度（甘えん坊は早い、ツンデレは遅いが上限が高い）

## 7. 新しい猫を増やす仕組み（v5更新）

### 基本ルート：成長解放
- 現在の猫が**若猫（15日以上）に育つ**と、2匹目を迎えられる
- 以降、レジェンドまで育てるごとに枠が増える
- レジェンド到達猫は「殿堂入り」として図鑑に永続記録

### サブルート1：レア報酬枠（v5新規）
- タスク完了時、確率（例：1日1回、5%）で**レア報酬枠**が出現
- タップで「新しいねこが出現！？」演出
- 出現する猫はランダム（レアリティ重みあり）

### サブルート2：にくきゅう召喚（v5新規）
- 30にくきゅうで指名召喚（任意の柄を1つ選択可）
- 50にくきゅうでレアリティ★3保証召喚
- 100にくきゅうで★4以上保証召喚

### サブルート3：シークレット条件
- シークレット猫は条件達成のみで出現（召喚不可）

## 8. タスク設計（整合性）

### 採点ロジック

完了「数」ではなく「達成率」で評価：

```
今日の評価 = 完了タスク数 ÷ 自分で設定したタスク数
```

| 達成率 | 餌グレード | 機嫌 | おさかな | にくきゅう |
|-------|-----------|------|---------|-----------|
| 100% | 高級おやつ | +30 | 完了数×2 + 5 | +1（任意） |
| 70%以上 | ウェットフード | +15 | 完了数×2 | 0 |
| 30%以上 | カリカリ | +5 | 完了数×1 | 0 |
| 0% | 餌なし | -20 | 0 | 0 |

### サボり防止

「設定数を減らせば100%になる」を防ぐため、過去7日の平均設定数の50%未満に減らした日は評価係数を0.7倍に下げる。

### タスクタイプ
- **ルーチンタスク**：毎日繰り返し（最大5件まで定型登録）
- **デイリータスク**：その日だけのもの
- **チャレンジタスク**：週1出現の高難度（達成で図鑑進捗ブースト + おさかな10）

### 餌システム
- 満腹ゲージは24時間で自然減少
- 毎日ログインしてタスクをやる動機

## 9. 成長設計（短縮版）

```
赤ちゃん猫（0-7日）→ 子猫（8-14日）→ 若猫（15-21日）→ 成猫（22-30日）→ レジェンド猫（31日〜）
```

達成率が低いと成長停止/退行。レジェンド到達後も**なかよしLv.**は伸び続けるため、長期動機が確保される。

3. クライアント側でアニメーションを再生

## 15. Gemini API活用

ユーザーが自分のAPIキーを設定して利用（運営側のAPI費用負担ゼロ）。

### 用途
| 機能 | 効果 |
|------|------|
| 猫との会話生成 | 毎日異なる挨拶・励まし・反応 |
| タスクへのアドバイス | パターンを見て改善提案 |
| 週次振り返り生成 | 1週間の達成パターンから分析コメント |
| シークレット猫の特別メッセージ | レアキャラごとに個性的なセリフ |
| なかよしLv.10刻みの特別な会話 | 親密度に応じた深い対話 |

### コスト感
- Gemini 2.5 Flash の無料枠：1日数百〜数千リクエスト
- 1ユーザー1日10リクエスト程度なら完全無料で運用可能

### APIキー設定
- ブラウザ版で楽に入力可（スマホでも可だが推奨はブラウザ）
- Supabase Vault でユーザーごとに暗号化保管
- フロントから直接叩かず、Edge Function 経由で呼ぶ（キー漏洩防止）

## 21. 自己レビュー：v5の懸念点

## 22. 管理画面・RBAC（v7：シンプル化）

### ロール構成（2種類のみ）

| ロール | 説明 | 付与方法 |
|--------|------|---------|
| `admin` | 管理者 | 管理画面でロール変更により昇格（初期は結パパさん） |
| `general` | 一般 | Googleログイン時にデフォルトで自動付与 |

招待・登録経路の概念は廃止。**Googleログインしたら誰でも一般ユーザーとして利用可**。
管理者を増やしたい場合は管理画面の「ユーザー」一覧でロール変更（自分自身は変更不可）。

### profiles テーブル（簡素版）

```sql
profiles
├── id              (uuid, Supabase Auth.users と連動)
├── email           (varchar)
├── display_name    (varchar)
├── role            (enum: 'admin' / 'general')
├── created_at      (timestamptz)
└── last_seen_at    (timestamptz)
```

### 管理画面のページ構成

| URL | 機能 |
|-----|------|
| `/admin` | ダッシュボード（総ユーザー数、ロール別内訳） |
| `/admin/users` | ユーザー一覧 + ロール変更（ドロップダウン、自分自身は変更不可） |

### ガード機構

- `/admin/*` 配下は **Server Component** で `role === 'admin'` チェック
- 違反時は **404相当**（管理画面の存在を秘匿）
- ロール変更APIは Server Action 側でも `requireAdmin()` で二重チェック

### RLS（Row Level Security）

- `profiles`：本人のみ閲覧、admin は全員閲覧可、本人のみ更新（admin は全員更新可）
- 全ゲームデータ（タスク、猫、通貨）：本人のみ、admin は閲覧のみ可
